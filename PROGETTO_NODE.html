PostgreSQL esterno · Solo lettura · Zero dipendenze ausiliarie

Approccio minimal · 17 aprile 2026

1Cosa costruiamo (e cosa NO)

✅ Dentro il container

  • App Next.js (frontend React + API routes)
  • Connessione al tuo PostgreSQL via DATABASE_URL
  • 6 viste pubbliche + dettaglio CUP
  • Unico processo Node, unica porta esposta

❌ FUORI dal container

  • PostgreSQL (è il tuo server, già esistente)
  • Nessun importer PDF/Excel
  • Nessun Redis, Celery, worker, cron
  • Nessun admin panel complesso
Il DB lo popoli tu/admin (manualmente, con script tuoi, o come preferisci). L'app Node legge e basta. Nessuna dipendenza di import, nessuna logica di parsing.

3Perché Next.js 14

Serve un framework Node che sia frontend + API in un solo processo. Le alternative valutate:

OpzioneProContro
Next.js 14 ⭐ SSR + API routes + static in un unico server Node. TypeScript nativo. SEO friendly. Community enorme. Leggermente più opinionated
Express + React SPA Massimo controllo, flessibile Serve configurare tante cose (routing, bundler, SSR)
NestJS + Angular Strutturato, simile a progetti tuoi esistenti (GCS) Due build separate, più pesante per il caso single-container
Remix / SvelteKit Moderni, veloci Ecosistema più piccolo
Raccomandazione: Next.js 14 App Router con TypeScript. Copre tutto (frontend + API + static + SSR) con un singolo npm run start. Perfetto per "container Node unico".

5Connessione al tuo PostgreSQL

Una sola variabile ambiente

# .env
DATABASE_URL="postgresql://bagnoli_ro:password@host.docker.internal:5432/bagnoli?schema=bagnoli"
NEXT_PUBLIC_SITE_URL="https://monitoraggio.analist24.it.com"

host.docker.internal permette al container di raggiungere il Postgres che gira sull'host fuori da Docker. In alternativa network_mode: host.

Prisma schema (mappa le tue tabelle)

// prisma/schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Cup {
  id             Int      @id @default(autoincrement())
  codice         String   @unique
  titolo         String
  tipo           String
  macro_area     String?
  tematica       String?
  attivo         Boolean  @default(true)
  tasks          Task[]

  @@map("cup")
  @@schema("bagnoli")
}

model Task {
  id             Int       @id @default(autoincrement())
  versione_id    Int
  cup_id         Int?
  activity_id    String?
  activity_name  String
  durata_giorni  Int?
  data_inizio    DateTime?
  data_fine      DateTime?
  inizio_actual  Boolean   @default(false)
  fine_actual    Boolean   @default(false)
  is_milestone   Boolean   @default(false)
  cup            Cup?      @relation(fields: [cup_id], references: [id])

  @@map("task")
  @@schema("bagnoli")
}
// ... altri modelli che riflettono il tuo schema_postgres.sql
Prisma non modifica il DB (usiamo prisma db pull dopo che hai creato lo schema). L'app è read-only: nessun CREATE, nessun ALTER, nessun INSERT.

6Esempio: API route e pagina

API route read-only (Next.js App Router)

// src/app/api/cruscotto/route.ts
import { NextResponse } from 'next/server';
import { prisma } from '@/lib/db';

export const revalidate = 300; // cache 5 min

export async function GET() {
  const [nCup, tasksTot, tasksCompl] = await Promise.all([
    prisma.cup.count({ where: { attivo: true } }),
    prisma.task.count(),
    prisma.task.count({ where: { fine_actual: true } }),
  ]);

  return NextResponse.json({
    n_cup: nCup,
    task_totali: tasksTot,
    task_completati: tasksCompl,
    pct_completamento: tasksTot ? Math.round(100 * tasksCompl / tasksTot) : 0,
  });
}

Pagina che la consuma (Server Component)

// src/app/page.tsx
import { GaugeSvg } from '@/components/GaugeSvg';
import { KpiCard } from '@/components/KpiCard';
import { prisma } from '@/lib/db';

export default async function HomePage() {
  const [nCup, tasksTot, tasksCompl] = await Promise.all([
    prisma.cup.count({ where: { attivo: true } }),
    prisma.task.count(),
    prisma.task.count({ where: { fine_actual: true } }),
  ]);
  const pct = tasksTot ? Math.round(100 * tasksCompl / tasksTot) : 0;

  return (
    <main className="hero">
      <h1>Programma Bagnoli-Coroglio</h1>
      <GaugeSvg value={pct} size={200} />
      <div className="kpi-grid">
        <KpiCard label="CUP attivi" value={nCup} />
        <KpiCard label="Task completati" value={tasksCompl} />
        <KpiCard label="Task totali" value={tasksTot} />
      </div>
    </main>
  );
}

7Dockerfile (multi-stage, 1 container)

# Dockerfile
FROM node:20-alpine AS base
RUN apk add --no-cache libc6-compat

# ---- deps ----
FROM base AS deps
WORKDIR /app
COPY package.json package-lock.json ./
COPY prisma ./prisma
RUN npm ci

# ---- build ----
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npx prisma generate
RUN npm run build

# ---- runtime ----
FROM base AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV PORT=3000

RUN addgroup --system --gid 1001 nodejs \
 && adduser  --system --uid 1001 nextjs

COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

USER nextjs
EXPOSE 3000
CMD ["node", "server.js"]

Usa output: 'standalone' in next.config.mjs → l'immagine finale pesa ~150-180 MB e non contiene sorgenti né node_modules di dev.

8Deploy — 3 comandi

# Sul server aaPanel:
cd /www/wwwroot/bagnoli-monitor

# 1. pull dal git
git pull origin main

# 2. build + run container
docker build -t bagnoli-monitor:latest .
docker stop bagnoli-monitor 2>/dev/null; docker rm bagnoli-monitor 2>/dev/null
docker run -d --name bagnoli-monitor \
  --restart unless-stopped \
  -p 127.0.0.1:3000:3000 \
  --add-host=host.docker.internal:host-gateway \
  --env-file /www/wwwroot/bagnoli-monitor/.env \
  bagnoli-monitor:latest

# 3. nginx aaPanel fa SSL + proxy → 127.0.0.1:3000
# (vhost gestito dal pannello o /www/server/panel/vhost/nginx/monitoraggio.conf)

Nginx esterno (aaPanel)

server {
    listen 443 ssl http2;
    server_name monitoraggio.analist24.it.com;

    ssl_certificate     /etc/letsencrypt/live/monitoraggio.analist24.it.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/monitoraggio.analist24.it.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
